# 新工程创建脚本使用说明

本文档说明如何使用工作区中的脚本，基于 `user_template` 模板快速创建新的 ESP32 用户工程，并自动生成基础调试配置。

## 1. 脚本位置

脚本位于：

```text
/workspaces/ESP32_dev/script/create_user_project.sh
```

模板目录位于：

```text
/workspaces/ESP32_dev/user_template
```

## 2. 脚本作用

该脚本会基于 `user_template` 自动创建一个新的用户工程，并完成以下处理：

- 复制模板目录到新的工程目录
- 自动跳过 `build/` 等不应带入新工程的内容
- 支持在创建时选择目标芯片
- 更新根目录 `CMakeLists.txt` 中的 `project(...)`
- 重命名模板中的主源文件和配置头文件
- 批量替换工程内的 `user_template` / `USER_TEMPLATE` 标识为新工程名
- 生成项目级 `.vscode/settings.json`、`.vscode/launch.json`、`.vscode/tasks.json`、`.vscode/extensions.json`
- 默认自动执行 `idf.py set-target` 与 `idf.py reconfigure`，让新工程尽量开箱即用
- 复制模板中的自动串口发现能力，适配 Docker / Dev Container + `usbipd attach`

## 3. 使用方法

在工作区根目录执行：

```bash
cd /workspaces/ESP32_dev
./script/create_user_project.sh [--target <chip>] [--port <auto|path>] <project_name>
```

例如：

```bash
./script/create_user_project.sh blink_demo
```

指定目标芯片示例：

```bash
./script/create_user_project.sh --target esp32s3 s3_demo
```

指定串口策略示例：

```bash
./script/create_user_project.sh --port auto c3_demo
./script/create_user_project.sh --port /host-dev/serial/by-id/your-device c3_demo
```

执行后会创建：

```text
/workspaces/ESP32_dev/blink_demo
```

## 4. 目标芯片选择

当前支持这些目标芯片：

- `esp32`
- `esp32s2`
- `esp32s3`
- `esp32c3`
- `esp32c5`
- `esp32c6`
- `esp32c61`
- `esp32h2`
- `esp32p4`

默认值：

```text
esp32c3
```

如果不指定 `--target`，脚本会按 `esp32c3` 创建工程。

## 5. 指定目标路径

如果你希望把新工程创建到自定义目录，也可以传入第二个参数：

```bash
./script/create_user_project.sh sensor_app /workspaces/ESP32_dev/projects/sensor_app
```

## 6. 工程名规则

`project_name` 需要满足以下规则：

- 只允许字母、数字、下划线
- 不能以数字开头

合法示例：

- `blink_demo`
- `sensor_app`
- `app1`

不合法示例：

- `123demo`
- `demo-app`
- `demo app`

## 7. 创建后会发生什么

假设执行：

```bash
./script/create_user_project.sh --target esp32s3 demo_app
```

脚本会自动完成类似这些变更：

- `project(user_template)` → `project(demo_app)`
- `main/user_template.c` → `main/demo_app.c`
- `main/include/user_template_config.h` → `main/include/demo_app_config.h`
- `USER_TEMPLATE_APP_NAME` → `DEMO_APP_APP_NAME`
- `USER_TEMPLATE_LOG_TAG` → `DEMO_APP_LOG_TAG`
- `Makefile` 中的 `TARGET` 默认值会被设置为 `esp32s3`
- 自动生成 `.vscode/` 下的基础构建与调试配置
- 默认自动执行 `idf.py set-target esp32s3` 和 `idf.py reconfigure`

## 8. 复制后的调试配置

脚本会在新工程内生成：

```text
.vscode/
├── extensions.json
├── launch.json
├── settings.json
└── tasks.json
```

这些配置提供：

- VS Code ESP-IDF 调试入口
- 项目级 `IDF_TARGET` 配置
- 项目级 `compile_commands.json` 路径
- 常用任务：`Build`、`Reconfigure`、`Flash`、`Monitor`、`Flash and Monitor`、`Doctor`、`Port`

## 9. `--port` 参数说明

脚本新增了：

```bash
--port <auto|path>
```

两种模式：

- `--port auto`
  - 创建工程时尝试自动探测当前唯一串口
  - 若成功，会把探测结果写入 `.vscode/settings.json` 的 `idf.port`
  - 若当前没有唯一串口，则不会写死错误值

- `--port /host-dev/serial/by-id/...`
  - 直接把你指定的串口路径写入 `.vscode/settings.json`
  - 适合你已经明确知道要使用哪个设备的情况

## 10. 自动串口发现

新工程会继承模板中的自动串口发现脚本：

```text
tools/detect_serial_port.sh
```

默认检测顺序：

1. `/host-dev/serial/by-id/*`
2. `/dev/serial/by-id/*`
3. `/host-dev/ttyACM*`
4. `/host-dev/ttyUSB*`
5. `/dev/ttyACM*`
6. `/dev/ttyUSB*`

这正是当前 Docker 容器内通过 `usbipd attach` 透传串口时最常见的设备分布。

查看当前自动发现结果：

```bash
make port
```

查看自动发现来源：

```bash
make doctor
```

其中会显示：

```text
PORT_SOURCE=host-dev-by-id
```

可能的来源值包括：

- `host-dev-by-id`
- `dev-by-id`
- `host-dev-ttyACM`
- `host-dev-ttyUSB`
- `dev-ttyACM`
- `dev-ttyUSB`

如果当前只检测到一个串口设备，则：

```bash
make flash
make monitor
make run
```

都可以直接使用，不必手写 `PORT`。

如果检测到多个串口，脚本会拒绝自动猜测，并提示你显式指定：

```bash
make run PORT=/host-dev/serial/by-id/your-device
```

## 11. 不会复制的内容

为了避免把模板构建残留带入新工程，脚本默认不会复制这些内容：

- `build/`
- `sdkconfig.old`
- `managed_components/`
- `dependencies.lock`

说明：

- 当前脚本会保留模板中的 `sdkconfig`
- 这样做的目的是继承模板已经验证过的 ESP32-C3 配置

## 12. 自动初始化与创建后建议操作

默认情况下，脚本会自动执行：

```bash
idf.py set-target <chip>
idf.py reconfigure
```

这样做的目的是：

- 让目标芯片配置与新工程一致
- 提前生成基础 `build/` 目录
- 让 VS Code / clangd 更快获得可用的编译数据库路径

如果你只想复制工程而不自动初始化，可以使用：

```bash
./script/create_user_project.sh --target esp32c6 --no-init demo_app
```

创建完成后，进入新工程目录执行：

```bash
cd /workspaces/ESP32_dev/demo_app
make reconfigure
make build
```

如果后续需要下载到开发板，还可以继续执行：

```bash
make flash
make monitor
```

或者：

```bash
make flash-monitor
```

## 13. 常见问题

### 13.1 提示模板目录不存在

请确认以下目录存在：

```text
/workspaces/ESP32_dev/user_template
```

### 13.2 提示目标路径已存在

说明你指定的新工程目录已经存在。可以：

- 换一个工程名
- 删除旧目录后重新执行
- 使用第二个参数指定新的目标路径

### 13.3 提示工程名不合法

请避免使用：

- 连字符 `-`
- 空格
- 中文
- 数字开头的名称

建议统一使用：

```text
小写字母 + 下划线
```

例如：

```text
wifi_logger
adc_demo
sensor_hub
```

### 13.4 自动初始化失败怎么办

如果脚本提示 `idf.py set-target` 或 `idf.py reconfigure` 失败，通常是因为：

- 当前不在已加载 ESP-IDF 环境的容器内
- `idf.py` 不在当前 `PATH` 中
- 工具链初始化尚未完成

这时可以进入工程目录后手动执行：

```bash
cd /workspaces/ESP32_dev/<your_project>
idf.py set-target <chip>
idf.py reconfigure
make build
```
